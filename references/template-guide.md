# 技术交底书模板指南

## 模板章节映射与生成策略

共 10 个核心章节，按生成策略分为三类：

### AI 直接生成类

**发明名称**
- 基于专利点描述，简洁反映技术主题
- 不含发明点细节，不限制保护范围
- 格式示例：一种 XXX 的方法/系统/装置

**技术领域**
- 适当范围，不过大不过细
- 不含发明点细节

**现有技术**
- 基于世界知识 + WebSearch 构建背景
- 客观描述，包含组成单元、连接关系、实现作用

**背景技术的缺陷**
- 对应用户选定的专利点，列出缺陷与原因
- 缺陷与原因一一对应
- 不添加商业宣传用语
- 限一个自然段

**有益的技术效果**
- 与方案关键部件/流程逐一对应
- 每个效果有理有据，不说空话

**关键点和保护点**
- 技术与传统方案比对
- 创新点分级标注：核心、较重要、一般重要
- **若用户提供了检索结果，必须包含"与核心现有技术的直接对比"子章节**
  - 对比格式：专利号 + 技术方案简述 + 明确区别点
  - 示例：中国一汽CN121579116A的预定义原子能力库为静态预定义集合，不支持用户行为驱动的自动更新；而本发明的迭代对象是场景库本体本身...
- 突出最精华部分

**可替代的技术方案**
- 从整体和核心特征两个层面发散
- 预防竞争对手回避设计

### AI 构建 + 用户确认类

**技术方案详述（核心章节）**
- 模型生成完整技术路径，产品经理确认后写入
- 必须包含：
  - 数据获取→处理→判断→输出的整体流程
  - 发明点环节深度展开
  - 流程图逻辑描述
  - 多种实现方式
  - 并列式和递进式实施方式
- 用户确认逻辑可行后，再写入交底书

### AI 生成描述 + 占位符提示类

**附图**
- 生成流程图和系统图的逻辑描述与文字说明
- 标注各部件名称
- 实际 DWG/VISIO 文件由专利代理人后续绘制
- 交底书中用 Mermaid 或文字框图占位

**其他补充信息**
- 可选，按需填写

### 用户手动填写类

**基础信息部分**
- 第一发明人姓名、工号、手机、邮箱、身份证
- 部门、申请类别、所属项目
- 其他发明人、专利申请人
- AI 以占位符形式保留，标注必填

## 三种构建模式

每个专利点展开前由用户选择：

### 模式 A：多候选选择
模型生成 2 到 3 个合理技术路径，产品经理选择最接近预期的。确认后按选定路径撰写交底书。

### 模式 B：单一最优构建
模型基于世界知识生成最合理的技术路径，产品经理确认或提出修改意见。确认后撰写交底书。

### 模式 C：先确认方向再展开
对话先聚焦关键决策点，如规则还是模型、XGBoost 还是神经网络。用户确认方向后，模型再展开为具体流程图和公式。

## 质量标准检查表

### 场景一（PRD 识别）
| 检查项 | 验证方式 |
|---|---|
| 五维扫描覆盖完整 | 输出中可见五个维度的扫描结果 |
| 每个候选点都有技术问题、方案、效果 | 输出中逐条可见 |
| 新颖性初判有依据 | 标注来源 |
| 明显不可专利的点被过滤 | 不出现纯业务逻辑、数学方法等不可专利内容 |
| 检索任务包可操作 | 包含关键词、检索系统、检索字段、反馈格式 |
| 检索任务仅限内部专利库 | 不派发沟通类任务 |

### 场景二（交底书撰写）
| 检查项 | 验证方式 |
|---|---|
| 严格匹配模板章节结构 | 输出含全部 10 个核心章节 + 基础信息占位符 |
| 发明名称 | 简洁反映技术主题，不含发明点细节 |
| 技术领域 | 范围适当，不过大不过细，不含发明点细节 |
| 现有技术 | 客观描述背景，包含组成单元、连接关系、实现作用 |
| 背景技术的缺陷 | 缺陷与原因一一对应，不添加商业宣传，限一个自然段 |
| 技术方案详述 | 数据获取→处理→判断→输出流程清晰；发明点环节深度展开；含流程图逻辑描述；提供多种实现方式；含并列式和递进式实施方式 |
| 有益的技术效果 | 关键部件/流程与效果逐一对应，有理有据 |
| 关键点和保护点 | 技术与传统方案比对，创新点分级，突出最精华部分；若用户提供检索结果，须含指名道姓的专利对比 |
| 可替代的技术方案 | 从整体和核心特征两个层面发散，预防竞争对手回避设计 |
| 附图 | 含流程图逻辑描述和系统结构说明，各部件标注名称 |
| 术语一致性 | 全文同一事物一种叫法 |
| 技术方案逻辑自洽 | 方案在逻辑上可行，符合技术常识 |
| 技术方案可做到 | 本领域技术人员按描述可以实施 |
| 不遗漏必填字段 | 模板要求的必填项都填写 |
| 三种模式可选 | 用户在展开前可选择模式，每种模式附简单说明 |
| 章节化增量撰写 | 每章独立写入文件，TodoWrite 跟踪进度，非一次性输出全部章节 |
| 阶段性交付物保存 | 专利挖掘结果和技术路径是否已保存为独立 md 文件 |
| 与检索到的核心现有技术直接对比 | 若用户提供了检索结果，关键点和保护点章节须含指名道姓的专利对比（专利号+技术方案简述+明确区别点） |
| 算法差异化表述 | 技术方案中的算法步骤须明确区别于常规聚类/统计分析/监督学习 |
| 检索提示词生成 | 若用户需要补充检索，须输出可直接粘贴到 AI 助手的检索提示词 |
| 交底书完成后询问修改 | 全部章节完成后须询问用户是否有修改意见 |

### 场景三（自审查）
| 检查项 | 验证方式 |
|---|---|
| 覆盖三性审查 | 新颖性、创造性、实用性逐项检查 |
| 模板字段完整性 | 10 个核心章节均有效填写，必填项无空缺 |
| 发明名称合规 | 不含发明点细节，未限制保护范围 |
| 技术领域适当 | 范围不过大不过细 |
| 背景技术客观 | 有理有据说明现有技术缺陷与原因 |
| 技术方案详述 | 流程完整、发明点展开充分、多种实现方式、并列递进式实施方式 |
| 技术效果对应 | 关键部件/流程与效果一一对应 |
| 保护点分级 | 核心、较重要、一般重要三级标注清晰 |
| 替代方案发散 | 从整体和核心特征两个层面覆盖 |
| 附图对应 | 文字描述与附图逻辑一致，部件名称统一 |
| 术语一致性 | 全文同一事物一种叫法 |
| 子 Agent 独立审查 | 是否调用了独立子 Agent 进行新颖性交叉验证 |
| 检索结果整合完整性 | 审查意见是否充分利用了用户提供的检索结果 |
| 审查后询问修改 | 审查结果输出后是否询问用户是否按建议修改 |
| 输出为机器可消费格式 | 审查结果可直接用于模型自修正 |
